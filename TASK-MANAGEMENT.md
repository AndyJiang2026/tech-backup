# 任务管理与交付规范

**制定日期**: 2026-04-01  
**制定原因**: 今日两次任务延误反思（承诺 30 分钟→实际 90 分钟；承诺 15 分钟→实际 36 分钟）

---

## 🎯 核心原则

1. **按时交付是第一优先级**
2. **逾期必须主动解释说明**
3. **双向沟通机制**
4. **目标一致，向下拆分**

---

## 📋 任务管理流程

### 一、任务接收阶段

```
用户需求 → 小马分析 → SMART 拆分 → 分配下属 → 确认承诺
```

**小马职责**：
- [ ] 分析任务类型和复杂度
- [ ] 选择合适的下属 Agent
- [ ] 用 SMART 原则拆分任务
- [ ] **明确截止时间**（精确到分钟）
- [ ] 确认下属理解任务目标

**下属 Agent 职责**：
- [ ] 确认收到任务
- [ ] 评估所需时间
- [ ] 承诺完成时间
- [ ] 如有困难立即提出

---

### 二、任务执行阶段

#### 双向沟通机制

| 时间节点 | 沟通内容 | 负责人 |
|---------|---------|-------|
| 任务启动时 | "已收到，预计 XX 分钟完成" | 下属 Agent |
| 25% 进度 | 正常：无消息；异常：立即报告 | 下属 Agent |
| 50% 进度 | 主动汇报进度（如任务>30 分钟） | 下属 Agent |
| 75% 进度 | "即将完成，预计 X 分钟内交付" | 下属 Agent |
| 100% 完成 | 交付成果 + 验收说明 | 下属 Agent |

#### 小马的跟进职责

| 时间点 | 行动 |
|-------|------|
| 承诺时间 30% | 检查进度（如任务>30 分钟） |
| 承诺时间 60% | 再次检查，确认能否按时 |
| 承诺时间 90% | 准备验收和交付 |
| 承诺时间 100% | **必须交付或解释** |

---

### 三、逾期处理机制

#### 逾期预警（承诺时间 80% 时）

```
如果下属未主动汇报 → 小马主动询问
如果确认无法按时完成 → 立即通知用户并说明原因
```

#### 逾期解释模板

```markdown
## ⚠️ 任务延误说明

**任务**: [任务名称]
**承诺时间**: [XX 分钟]
**预计完成**: [XX 分钟]
**延误原因**: 
1. [具体原因 1]
2. [具体原因 2]

**改进措施**:
- [措施 1]
- [措施 2]

**预计交付时间**: [具体时间]

非常抱歉给您带来不便，我会全程跟进确保尽快完成！
```

#### 逾期后的行动

1. **立即通知用户**（不要等用户来问）
2. **说明具体原因**（不找借口）
3. **给出新的承诺时间**（确保能完成）
4. **全程跟进直到完成**
5. **事后复盘改进**

## 🎓 下属 Agent 管理

### 时间观念培养

**每次任务分配时强调**：
```
1. 明确截止时间（精确到分钟）
2. 确认对方理解并承诺
3. 说明逾期的后果（影响团队信誉）
4. 要求主动汇报进度
```

### 目标向下拆分

**原则**：
- 大目标 → 子目标 → 具体任务
- 每个子任务都有明确验收标准
- 子任务完成时间之和 < 总时间（留 buffer）

**示例**：
```
总任务：审查合同并生成 Word（30 分钟）
├─ 法务审查：15 分钟
├─ 修改建议：5 分钟
├─ Word 生成：5 分钟
├─ 验收检查：3 分钟
└─ Buffer: 2 分钟
```

### 双向沟通要求

**下属必须做到**：
1. 收到任务立即确认
2. 遇到困难立即报告（不要硬扛）
3. 进度异常立即通知
4. 完成后主动交付（不要等询问）

**小马必须做到**：
1. 任务分配清晰明确
2. 主动跟进进度（不要只等通知）
3. 及时协调资源解决问题
4. 验收后及时交付给用户

### 每周检查

1. 任务按时完成率是否提升？
2. 逾期解释是否及时？
3. 下属 Agent 的沟通是否改善？
4. 任务拆分是否合理？

### 改进措施

根据复盘结果调整：
- 任务分配策略
- 时间估算方法
- 沟通频率
- 跟进机制

## 💡 关键教训（今日）

### 教训 1：被动等待通知
**问题**：启动子任务后完全依赖系统自动通知
**改进**：建立主动检查机制（30%/60%/90% 时间点）

### 教训 2：没有中间汇报
**问题**：用户不问就不知道进度
**改进**：50% 进度必须主动汇报

### 教训 3：逾期不解释
**问题**：超时后等用户发现才说明
**改进**：80% 时间确认无法完成时立即解释

### 教训 4：目标拆分不清晰
**问题**：下属不知道具体要交付什么
**改进**：SMART 原则 + 验收标准 + 交付格式
